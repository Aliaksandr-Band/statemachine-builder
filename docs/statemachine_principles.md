**характеристики библиотеки конечного автомата:**
 - используется builder api для описания конфигурации конечного автомата
 - конфигурация позволяет описать конечный автомат в соответствии с UML State Machine
 - конфигурация включает следующие элементы:
   - тип сообщения с которым работает конечный автомат (сообщения подающиеся на вход в конечный автомат состоят их generic-события и generic-payload. Generic-событие, как правило String или Enum. Payload - объект любого типа)
   - автомат поддерживает понятие отложенных сообщений (deferred), определённые сообщения могут накапливаться в определенном состоянии с последующей их обрабткой при смене состояния
   - набор состояний, с указанием стартового и не менее одного финальных состояний
   - набор внешних переходов между состояниями по конкретным сообщениям
   - набор внутренних переходом для конкретного состояния
   - выбор подходящего перехода по условиям прохождения проверки guard
   - определение действий (transitionActions) для внешних и внутренних переходов, для выхода из состояния и входа в него (exitActions, enterActions)
   - определение асинхронных действия после внешних и внутренних переходов (asyncActions)
   - у конечного автомата есть контекст в виде карты key-value, он обеспечивает хранение дополнительного контекста во время работы автомата, экшены автоматат и guards будут иметь доступ к контексту автомата
   - потокобезопасность доступа к контекству конечного автоматат обесечивается реализацией ConcurrentHashMap
 - к конфигурации применяются правила валидации:
   - состояния должны быть заданы, должны присутствовать минимум два состояния - начальное и конечное
   - переходы должны быть заданы между известными состояниями
   - топология валидна если все состояния, кроме начального, достижимы из начального. Из каждого не-финального состояния должен существовать хотя бы один путь до финального состояния.
   - при наличии асинхронных действий (asyncActions) должен быть указать ExecutorService
 - помимо внутренних состояний, которые задает конфигурация автомата, сам конечный автомат как процессор может находиться в четырех состояниях, которые определяют режим его использования:
   - READY - конечный автомат сконфигурирован и построен, возможно был сброшен в исходное состояние методом reset(), но не было вызова метода start(), сообщения не буду обрабатываться
   - RUNNING - конечный автомат после вызова методов start()/resume() и до попадания в финальное состояние заданное конфигурацией, сообщения обрабатываются
   - PAUSED - конечный автомат на паузе, после метода pause(), сообщения игнорируются, метод обработки сообщения вернет false, можно вернуть в RUNNING при помощи resume()
   - STOPPED - конечный автомат остановлен после достижения финального состояния либо вызова метода stop(), сообщения больше не принимаются, возобновление работы возможно вызовом reset()
     после возобнавления работы, предыдущий контекст работы конечного автомата очищается.

**концепции определённости конечного автомата**
 - логика должна гарантировать атомарность смены состояний автомата, т.е. автомат всегда должен быть в конкретном состоянии в зависимости от обработанных сообщений, автомат не может быть без состояния
 - если возникает ошибка в части присоединенного пользовательского кода (guards, transitionActions, stateActions, asyncActions), пользовательский код должен быть не замедлительно уведомлен об этом
 - у точек подключения логики к конечному автомату (guards, transitionActions, stateActions, asyncActions) должны быть callback методы обработки исключений
 - если в процессе перехода (цепочка Событие → guard → [exit actions] → [transition action] → [entry actions]) в синхронном коде (guards, transitionActions, stateActions) возникло исключение, то переход не будет осуществлен
 - нахождение конечного автомата в конкретном состоянии гарантирует что все предыдущие связанные синхронные действия прошли успешно
 - ошибки возникающие в asyncActions не будут влиять на осуществление перехода
 - слушатели действий конечного автомата получают уведомления в асинхронном режиме
 - слушатели будут получать уведомления о следующих событиях в конечном автомате - запуск/смена состояния/отказ в обработке сообщения/накопление deferred сообщения/остановка, 
 - исключения и ошибки в слушателях конечного автомата не будут влиять на смену его состояний
 - отличие внутренних переходов от внешних заключается в следующем:
   - Не вызывает exit / entry действия текущего состояния
   - Если action внуреннего перехода выбрасывает исключение — внешнее состояние машины остаётся прежним, информация об исключении передается в callback экшена

**дополнительные замечания - ошибки и асинхроность**
 - конечный автомат не обеспечивает rollback действий, которые были выполнены до возникновения исключения, но обеспечивает вызов соотвествующих callback методов
 - конечный автомат используется синхронизацию методов обработки входящих сообщений
 - конечный автомат обеспечивает уведомление слушателей в отдельном потоке в асинхронном режиме
 - конечный автомат обеспечивает выполнение асинхронных действий в отдельном потоке
 - исключения возникающие в асинхронных операциях не влияют (не останавливают) выполенение следующего асинхронного действия
 - возможность для реакции на исключения в асинхронных операциях обеспечивается через callback механизм
 - исключения в асинхронных операциях перехватываются и передаются в callback точки как Throwable
 - вслучае остановки конечного автомата ExecutorService не останавливается, управление сервисом ответственность пользовательского кода. При этом новые задачи поступать в сервис не будут, но старые накопленныем будут завершены

**схема поддержки deferred сообщений**
 - отложенные сообщения храняться в очереди на уровне конечного автомата
 - отложенные сообщения описываются для состояния во время конфигурации
 - отложенные сообщения обрабатываются по правилу FIFO
 - отложенные сообщения обрабатываются после завершения перехода, т.е. после цепочки - Событие → guard → [exitActions] → [transitionActions] → [entryActions]
 - очередь из отложенных сообщений может приводить цепочке переходов, если каждое следующее состояние может работать со следующим отложенным сообщением, но бесконечной рекурсии возникнуть не может
 - отложенные сообщения которы беруться в обработку извлекаются из очереди и больше обрабатываться не будут, если только не станут снова отложенными в новом состоянии, тогда помещаются в конец очереди
 - отложенные сообщения могут приводить как внешним так и к внутренним переходам (без смены состояния)

**недоработки или компромисы**
 - для асинхронных слушателей уведомлений от конечного автомата не гарантируется соблюдение последовательности уведомлений, т.к. уведомление работает через общий executor
 - на текущий момент не состояний FAULTED / ERROR т.к. нет четкого понимания сценариев в чем их польза и что с ними делать
