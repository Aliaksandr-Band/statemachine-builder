# Концептуальное описание библиотеки «Lightweight Zero-Dependency State Machine»

## 1. Философия и ключевые характеристики

Библиотека предоставляет реализацию конечного автомата (Finite State Machine) в соответствии с концепциями UML State Machine. Главные принципы:

- **Zero dependencies** – используется только Java Standard Library (пакет `java.util.concurrent` для асинхронной работы).
- **Декларативная конфигурация** через Builder API.
- **Гарантия атомарности синхронных переходов** (на уровне одного вызова `sendMessage`).
- **Чёткое разделение синхронной и асинхронной логики**, а также жизненного цикла самого процессора.
- **Предсказуемая обработка ошибок** с уведомлением через callback’и и без магических восстановлений.

## 2. API Конфигурации (Builder)

Конфигурация строится с помощью fluent-интерфейса и включает следующие элементы:

| Элемент | Описание |
| :--- | :--- |
| **Тип сообщения** | Обобщённое событие `E` (обычно `String` или `Enum`) и payload `P` любого типа. |
| **Состояния** | Набор состояний типа `S` (обычно `String` или `Enum`). Обязательно указывается **одно стартовое** и **минимум одно финальное** состояние. |
| **Внешние переходы** | Переход между различными состояниями, инициируемый событием `E`. Может иметь `guard` и `action`. |
| **Внутренние переходы** | Переход, не меняющий состояния, инициируемый событием `E`. Может иметь `guard` и `action`. |
| **Guard (условие)** | Предикат, получающий контекст и payload события. Переход выполняется, только если guard возвращает `true`. При нескольких подходящих переходах используется порядок объявления (первый совпавший с истинным guard). |
| **Действия (Actions)** | Пользовательский код, выполняемый синхронно: `exitActions` (при выходе из состояния), `transitionAction` (в процессе перехода), `entryActions` (при входе в состояние). |
| **Асинхронные действия** | `asyncActions`, выполняемые в отдельном потоке после успешного завершения синхронной части перехода. |
| **Контекст** | Потокобезопасное хранилище данных вида `Map<String, Object>` (реализация `ConcurrentHashMap`), доступное во всех действиях и guard’ах. |
| **Отложенные события (Deferred)** | События, которые в указанном состоянии не обрабатываются немедленно, а помещаются в общую очередь и будут обработаны при смене состояния. |

### 2.1 Правила валидации конфигурации
При вызове `build()` выполняются следующие проверки:
- Задано хотя бы одно начальное и одно конечное состояние.
- Все переходы ведут только между состояниями, объявленными в конфигурации.
- **Достижимость:** Все состояния (кроме начального) достижимы из начального. Из любого не-финального состояния существует путь до хотя бы одного финального.
- Если в конфигурации присутствуют `asyncActions`, то в билдер должен быть передан экземпляр `ExecutorService`.

## 3. Жизненный цикл процессора конечного автомата

Помимо бизнес-состояний (`S`), сам объект автомата может находиться в одном из четырёх мета-состояний, определяющих его готовность обрабатывать сообщения.

| Состояние | Описание | Допустимые операции |
| :--- | :--- | :--- |
| **READY** | Конфигурация загружена, автомат создан или был сброшен вызовом `reset()`. Ожидает `start()`. | `start()`, `reset()` |
| **RUNNING** | Активная обработка сообщений. | `sendMessage()`, `pause()`, `stop()` |
| **PAUSED** | Временная приостановка. Вызов `sendMessage()` немедленно возвращает `false`. | `resume()`, `stop()` |
| **STOPPED** | Работа завершена (достигнуто финальное состояние или вызван `stop()`). | `reset()` |
| **FAULTED** | **Важно:** Возникает при неперехваченном исключении в синхронных блоках `exitActions` или `entryActions`. Автомат переходит в неопределённое состояние и больше не может обрабатывать события. | Только `reset()` |

> **Примечание по FAULTED:** Состояние вводится для соблюдения инварианты «автомат не может быть без состояния». Так как после частичного выполнения `exitActions` откат невозможен, единственный безопасный выход — объявить состояние повреждённым и потребовать явного сброса.

## 4. Концепции определённости и обработка ошибок

### 4.1 Порядок синхронного перехода (атомарная фаза)
Каждый переход (внешний или внутренний) выполняется в рамках синхронизированного метода `sendMessage` в строгой последовательности:

```
Событие (E, P) 
→ guard (если есть) 
→ [exitActions]          // только для внешнего перехода
→ [transitionAction] 
→ [entryActions]         // только для внешнего перехода
→ изменение состояния
```

**Правила обработки исключений на этой фазе:**
- Если исключение выброшено в **guard** или **transitionAction** (для любого перехода) либо во **внутреннем action** — переход **полностью отменяется**, текущее состояние не меняется.
- Если исключение выброшено в **exitActions** или **entryActions** — гарантировать целостность состояния невозможно, поэтому машина переходит в мета-состояние **FAULTED**.
- Библиотека **не выполняет автоматический rollback** действий. Ответственность за компенсацию побочных эффектов лежит на пользовательском коде через предоставляемые callback’и ошибок.

### 4.2 Асинхронные действия и слушатели
- **asyncActions** запускаются в переданном `ExecutorService` **после** успешного завершения синхронной части перехода и **параллельно** с началом обработки очереди отложенных событий (см. п. 5).
- Исключения в `asyncActions` **не влияют** на состояние автомата и не прерывают выполнение других асинхронных задач.
- Исключения из `asyncActions` и слушателей перехватываются и передаются в соответствующие **error-коллбэки** (например, `onAsyncError`).

### 4.3 Слушатели (Listeners)
- Уведомления слушателям (`onStateChanged`, `onEventIgnored`, `onDeferred`, `onStart`, `onStop`) отправляются **асинхронно** через тот же `ExecutorService`.
- **Порядок доставки уведомлений не гарантируется** (из-за использования общего пула потоков).
- Исключения в слушателях подавляются и логируются (если подключён логгер), но **не влияют** на работу автомата.

## 5. Семантика отложенных событий (Deferred Events)

### 5.1 Хранение и обработка
- Отложенные события хранятся в общей очереди (FIFO) на уровне экземпляра автомата.
- Событие помечается как `deferred` для конкретного состояния в конфигурации. Если автомат находится в этом состоянии и получает такое событие, оно добавляется в очередь вместо немедленной обработки.
- Обработка очереди запускается **строго после** завершения синхронной фазы перехода (`entryActions`), но **не ждёт** завершения `asyncActions`, запущенных этим переходом.

### 5.2 Правила потребления очереди
1. Обработка идёт по одному событию за раз (с сохранением внешней синхронизации `sendMessage`).
2. Событие извлекается из очереди. Если в текущем состоянии это событие **снова** помечено как `deferred`, оно возвращается в **конец** очереди.
3. Отложенное событие может инициировать как внешний переход (со сменой состояния), так и внутренний.
4. **Предупреждение для пользователей:** Внутренний переход, который всегда возвращает событие в очередь (как `deferred`), приведёт к бесконечному циклу, если не использовать `guard` для его однократного выполнения или не изменить контекст.

## 6. Дополнительные замечания и ограничения

- **Синхронизация:** Публичные методы управления (`sendMessage`, `start`, `pause`, `resume`, `stop`, `reset`) синхронизированы на уровне экземпляра для обеспечения атомарности в многопоточной среде.
- **Управление ExecutorService:** Библиотека **не управляет** жизненным циклом переданного `ExecutorService`. Вызовы `stop()` или `reset()` прекращают отправку новых задач, но уже запущенные асинхронные задачи будут завершены.
- **Контекст:** Хотя `ConcurrentHashMap` обеспечивает базовую потокобезопасность операций put/get, пользователь должен самостоятельно обеспечивать атомарность составных операций (например, `get` → модификация → `put`).

## 7. Стратегия версионирования и границы функциональности

Для предотвращения превращения библиотеки в громоздкий фреймворк вводится чёткое разделение функциональности по версиям.

### 7.1 Версия 1.0 (MVP – Minimal Viable Product)

**Цель:** Предоставить строгую, атомарную синхронную машину без внешних зависимостей, пригодную для использования в однопоточных средах (включая GWT-совместимые) и простых серверных приложениях.

**Объём функциональности:**
- [x] Builder API и валидация топологии.
- [x] Внешние переходы (`guard`, `transitionAction`, `exit/entry`).
- [x] Внутренние переходы (без `exit/entry`).
- [x] Потокобезопасный контекст.
- [x] Жизненный цикл: `READY`, `RUNNING`, `STOPPED`.
- [x] Состояние `FAULTED` при критических ошибках в `exit/entry`.
- [x] Callback’и ошибок для синхронных действий.
- [ ] **Исключено:** Асинхронные слушатели.
- [ ] **Исключено:** `asyncActions`.
- [ ] **Исключено:** Отложенные события (`deferred`).
- [ ] **Исключено:** Состояние `PAUSED`.

**Почему именно так?**  
Версия 1.0 должна быть идеально надёжной и простой. Отсутствие асинхронности и deferred-очередей исключает сложнейшие сценарии гонок данных и позволяет отточить ядро библиотеки.

### 7.2 Версия 2.0 (Целевое полное описание)

**Цель:** Реализовать полный набор возможностей, описанных в данном документе, оставаясь при этом легковесной и zero-dependency.

**Добавления к версии 1.0:**
- [x] Поддержка `asyncActions` с пользовательским `ExecutorService`.
- [x] Асинхронные слушатели жизненного цикла и переходов.
- [x] Полноценная очередь `deferred` событий (FIFO) с семантикой, описанной в разделе 5.
- [x] Состояние `PAUSED` в жизненном цикле процессора.
- [x] Уточнённые гарантии поведения при останове/сбросе с активными асинхронными задачами.

### 7.3 Что останется за рамками (Anti-goals)
Для сохранения идентичности «легковесной библиотеки» следующие возможности **не будут реализованы никогда**:
- Иерархические (вложенные) состояния.
- Ортогональные регионы (параллельные состояния).
- Псевдосостояния `History` (глубокое/поверхностное).
- Встроенная поддержка персистентности состояния (сериализация в БД).
- Автоматическое управление пулом потоков.
